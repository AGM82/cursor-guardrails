Everything you need to start a new project or adopt guardrails on an existing one. All prompts are pre-filled with your template path.

What cursor-guardrails is

A reusable project template that makes good engineering habits automatic. Written rules are advice; this template puts the real weight into checks a machine enforces and nobody can skip. github.com/AGM82/cursor-guardrails

The 7 guardrail tiers

TierMechanismLives inType
0 — WorkflowPlan Mode (Shift+Tab); plans saved to .cursor/plans/CursorDiscipline
1 — AdvisoryRules in .cursor/rules/*.mdc; cross-tool summary in AGENTS.mdRepoAdvisory
2 — ToolchainTypeScript strict, ESLint (security + a11y), Prettier, commitlint, testsRepoDeterministic
3 — RuntimeCursor hooks gate shell commands and secret reads; audit logRepoDeterministic
4 — AutomationHusky + lint-staged + gitleaks pre-commit; CI (typecheck, lint, test, build, audit, secret scan, SAST)Repo/CIDeterministic
5 — WorkflowsSlash commands: /review, /audit, /pr, /update-deps, /guardrail-upgradeRepoWorkflow
6 — ReviewAgent Review, Bugbot on PRs, TDD for logic-heavy workCursor/CISafety net

The 5 slash commands

CommandWhat it does
/reviewRun checks and review staged changes; report by severity before fixing anything
/auditWhole-repo, point-in-time health audit — not scoped to a diff
/prConfirm checks pass, write a Conventional Commit, push, and open a pull request
/update-depsUpdate dependencies one at a time, re-testing after each
/guardrail-upgradeLayers 0–7: bootstrap → profile → gap analysis → upgrade → code audit → governance

Hard stops (always)

  • No secrets, credentials, or real client/personal data in source, logs, or commits
  • Validate input at every trust boundary; use parameterised queries only
  • Never log personal information
  • Do not modify tests to make failing code pass
  • Do not disable a linter or type rule to silence an error — fix the cause
  • CI enforces coverage thresholds (lines ≥ 80%, functions ≥ 80%, branches ≥ 70%) — do not merge code that drops below them
  • Never commit or push directly to main — use a feature branch and /pr

Start a new project from the starter

Before you start: you need a free GitHub account and Cursor installed. No other software is required — Cursor can clone repositories for you without a separate Git install.

New projects use the starter, not this hub. Create your repo from the thin cursor-guardrails-starter (Layers 1–5 + demo only). Do not "Use this template" on cursor-guardrails itself — the hub ships playbook, Throughline, and AI-review scaffolding that a product repo shouldn't carry. "Use this template" intentionally creates an independent repo with no live link back. To pull in future guardrail improvements later, keep a separate reference clone — a plain git clone of the hub (cursor-guardrails), see the Adopt on existing project tab — and run /guardrail-upgrade inside this project. Same tool, same result as adopting on an existing project.

#StepDetail
1Use this template (starter)Go to github.com/AGM82/cursor-guardrails-starter → Use this template → Create a new repository. (Not the cursor-guardrails hub — that's the reference clone, not the starter.)
2Clone it to your computerIn Cursor: Command Palette (Ctrl+Shift+P / Cmd+Shift+P) → Git: Clone → paste your new repo's URL → pick a folder. (Or, if you're comfortable with a terminal: git clone <url>.) This downloads the repo from GitHub onto your machine — you can't skip this step.
3Open in CursorFile → Open Folder → pick the folder you just cloned → Trust the workspace
4Match Node versionNode.js is the runtime this project's tools (linting, tests, build) run on. Open a terminal (Terminal → New Terminal) and run nvm use if you have nvm installed — otherwise, install Node 20 LTS from nodejs.org directly and skip this command.
5Installnpm install in the same terminal — installs tools and activates Git hooks
6VerifyRun npm run typecheck, then npm run lint, then npm run test, one at a time (on Windows, plain PowerShell can't chain them with && — PowerShell 7+ can). CI also runs test -- --coverage; if anything fails, ask the Cursor agent to fix it before continuing.
7Fill in contextOpen .cursor/rules/90-project-context.mdc and replace every <PLACEHOLDER> — or use the "Generate 90-project-context.mdc" prompt in the Prompt Library tab and let the AI draft it from a plain description of your project
8Commit and pushCommit your changes and push to GitHub — this triggers the first CI run, which step 9 needs to have completed at least once
9Enable branch protectionWait for the CI run from step 8 to go green under the Actions tab (its 3 checks won't be selectable until then), then GitHub Settings → Branches → Add branch ruleset → require all 3 checks on main. Note: on a free personal GitHub account, rulesets only enforce on public repositories.

Adopt on an existing project — quickstart (Path A, direct)

Two folders are involved: the project you want to upgrade (call it Project X) and a separate local clone of the cursor-guardrails repo itself — its reference clone. The reference clone is only there so the agent has something to copy files from — you never open it or work inside it.

Don't have a reference clone yet? Create it once, anywhere outside Project X:
git clone https://github.com/AGM82/cursor-guardrails
Use a plain clone — not the "Use this template" button from the other tab. That button is for starting a brand-new project; it makes a disconnected copy with no link back to this repo, so it could never receive updates. A reference clone stays linked, so /guardrail-upgrade can refresh it.

Staying current: the version shown at the top of this page is always the latest published template version. Your local reference clone only reflects that once you pull it — but you don't have to do this by hand: the Bootstrap prompt below and the /guardrail-upgrade command both refresh it automatically (git pull --ff-only) at the start of every run, before copying anything from it.

#StepDetail
1Fill in the reference clone pathAt the top of this page, set "Reference clone path on this machine" to where you cloned cursor-guardrails (not Project X). The prompt below is pre-filled from it.
2Paste the promptOpen Project X in Cursor, start a new Agent chat, and paste the Bootstrap prompt below. It works day zero — nothing needs to be in place first, and it refreshes your reference clone before using it.
3Answer 3 quick questions, then approve layersThe agent asks about the project's type, risk level, and existing toolchain, then recommends which layers to apply based on the answers — not everything, and not identical for every project. Reply all instead if you'd rather apply everything in one pass. (Skipped automatically if a Throughline guardrail-prescription.json is found — see the Via Throughline tab.)

Bootstrap prompt — paste into Cursor Agent chat

Upgrade this project from the cursor-guardrails template.

Template path:
TEMPLATE_PATH_PLACEHOLDER

Follow this workflow:
1. Verify the template path is a git clone and refresh it so nothing below reads a stale snapshot: run `git -C "TEMPLATE_PATH" pull --ff-only origin main`. If the folder does not exist or is not a git repo, tell me to run `git clone https://github.com/AGM82/cursor-guardrails "TEMPLATE_PATH"` once (not GitHub's "Use this template" button — that can't receive updates), then stop and wait for me to confirm before continuing. If the pull fails for another reason, compare TEMPLATE_PATH/.cursor/guardrail-version against the published version at raw.githubusercontent.com/AGM82/cursor-guardrails/main/.cursor/guardrail-version, warn me if it's behind, and continue anyway.
2. Copy .cursor/commands/ from the template into this project (create the folder if it does not exist).
3. Capture a pre-upgrade baseline: run typecheck, lint, build, and test (skip any that do not exist). Save output to .cursor/guardrail-baseline.log.
4. Commit everything as: chore: snapshot before guardrail upgrade
5. Check for guardrail-prescription.json (project root, then .cursor/) — if found, use its tier and requiredLayers directly and skip the questions below. Otherwise ask me 3 quick questions to build a project profile: project type (frontend-ui / backend-api / full-stack / library-or-cli / script-or-prototype), risk level (Low / Medium / High), and whether I already have my own ESLint/Prettier/tsconfig setup. Use TEMPLATE_PATH/guardrail-layers.json -> projectProfiles and riskTiers to turn my answers into a recommended layer set.
6. Compare both projects and show a gap analysis table by layer, alongside the recommended layer set from step 5.
7. Ask me which layers to apply before changing anything.
8. Never overwrite src/, package.json, or tsconfig.json — merge only.
9. After the approved layers: run a code compliance audit (Layer 6) and report Blocker / Should-fix / Nit findings. Fix blockers before finishing.
10. Show the Layer 7 governance checklist.

Once adopted, this is repeatable: run /audit any time for a whole-codebase health check, and re-run /guardrail-upgrade later to detect drift against a newer template version.

You don't need to read this to get started — it's what the prompt above walks the agent through, for when you want to see what it's about to do or hold off on a specific layer.

Power users only: a PowerShell one-liner can copy Layer 0 (just the .cursor/commands/ folder) without an agent chat — you'd still type /guardrail-upgrade afterward for everything else. Most people should just use the Bootstrap prompt above.

$template = "TEMPLATE_PATH_PLACEHOLDER"
$dest = ".cursor\commands"
New-Item -ItemType Directory -Force -Path $dest | Out-Null
Copy-Item "$template\.cursor\commands\*" $dest -Recurse -Force
Write-Host "Commands bootstrapped. Now type /guardrail-upgrade in Cursor chat."

Adopt via Throughline — quickstart (Path B)

What this adds over Path A: Path B is Path A with one difference — instead of answering /guardrail-upgrade's 3 profiling questions yourself, Throughline classifies the project deterministically (its Schedule A risk rubric) and hands the resulting tier + required layers to /guardrail-upgrade as a small file. Same executor, same reference clone, same result — just an auditable classification instead of a manual answer.

Use this when: the project needs a documented risk tier, governance evidence, or the CTO-facing lifecycle story — not just guardrails applied. Otherwise, the Adopt on existing project tab alone is enough.

#StepDetail
1Classify in ThroughlineDescribe Project X in Throughline. Its deterministic classifier returns a risk tier (Low / Medium / High) and, via the vendored guardrail-layers.json, the required layers for that tier.
2Download the prescriptionFrom Throughline's Build Package, download guardrail-prescription.json.
3Drop it into Project XSave it at the project root (or .cursor/).
4Run /guardrail-upgradeBootstrap first via the Adopt on existing project tab if this is day zero. The command detects the prescription automatically, warns if it's stale against the reference clone's version, and uses its tier and layers directly — no profiling questions asked.
5Approve layers and verifySame as Path A — review the gap analysis, approve, then run npm run typecheck / lint / test.

Guardrail files (rules, hooks, CI, toolchain) always come directly from the reference clone via /guardrail-upgrade — never through Throughline. Throughline only ever supplies the prescription (which layers, and why). Full detail: docs/connect-guardrails.md and docs/guardrail-prescription.md.
